# 使用说明

本文档说明 `sili-jian-orchestrator` 的推荐使用方式。

## 一、技能定位

这个技能是“司礼监 / 总调度 Agent”技能，不是普通编码技能。

它的职责是：

- 接收需求
- 识别项目
- 判断场景
- 建立与维护治理体系
- 组织方案
- 组织执行
- 组织测试
- 组织审查
- 组织恢复
- 维护状态、交接、门禁、报告与步骤快照

因此，调用它时的正确姿势通常是：

**先接管、先盘点、先治理，再决定是否执行。**

## 二、适合触发的场景

以下场景推荐使用本技能：

1. 接管一个做到一半的项目
2. 恢复一个暂停中的项目
3. 给旧项目补治理结构
4. 先把模糊需求整理成方案，而不是直接开发
5. 把新增功能纳入现有治理体系
6. 长期维护一个大项目

以下场景通常不需要使用本技能：

1. 单纯改一个按钮
2. 修一个小 bug
3. 写一个普通小接口
4. 一次性的小改动且不需要治理流程

## 三、首次调用建议

第一次调用时，建议先跑首次启用引导。

推荐提示词：

```text
使用 $sili-jian-orchestrator 对当前目录执行首次启用引导。

要求：
1. 先不要直接开发
2. 先判断当前目录是技能目录、项目目录还是未知目录
3. 检查 OpenClaw 中司礼监与同级部门 agent 是否已就绪
4. 如果当前目录是项目目录，再继续检查治理状态、tests、workflows、状态文件、恢复入口
5. 输出首次引导结果，至少包含：
   - 当前目录模式
   - peer-agent 是否就绪
   - 当前环境意味着什么
   - 最安全的下一步
   - 建议我下一条输入什么提示词
6. 如果当前目录已经是项目目录，再继续输出首轮接管结果
7. 在完成上述检查前，不要进入实现阶段
```

## 四、在技能目录中测试

如果你当前位于技能目录本身，可以使用：

```text
使用 $sili-jian-orchestrator 对当前目录执行首次启用引导。

当前目录大概率是技能封装目录，不是业务项目目录。
要求：
1. 不要直接开发
2. 先判断当前目录模式
3. 检查 OpenClaw 中司礼监、内阁、都察院、六部 agent 是否已就绪
4. 如果这里是技能目录，只输出首次引导结果，不要创建 ai/、tests/、workflows/ 等项目治理文件
5. 输出：
   - 当前目录模式
   - peer-agent 就绪情况
   - 当前环境说明
   - 最安全的下一步
   - 建议我下一条输入什么提示词
```

## 五、在真实项目中接管

如果你已经进入真实项目目录，建议使用：

```text
使用 $sili-jian-orchestrator 接管当前项目，并先执行首次启用引导，不得直接开发。

要求：
1. 先判断当前目录是否为项目根目录
2. 检查 OpenClaw 中司礼监、内阁、都察院、六部 agent 是否已就绪
3. 执行首轮接管检查
4. 检查项目识别、ai/、tests/、workflows/、交接入口、状态文件、审批规则、测试规则、工作流模板、步骤快照、任务卡模板
5. 输出首次引导结果，至少包含：
   - 当前目录模式
   - peer-agent 就绪情况
   - 当前环境说明
   - 最安全的下一步
   - 建议我下一条输入什么提示词
6. 再输出首轮接管结果，至少包含：
   - 当前项目识别结果
   - 当前属于哪种场景
   - ai/ 是否齐全
   - tests/ 是否齐全
   - workflows/ 是否齐全
   - 是否存在显式状态机
   - 是否存在最近步骤快照
   - 缺失的治理文件
   - 缺失的测试体系
   - 缺失的 workflow 模板
   - 缺失的恢复机制
   - 需要先创建的文件和目录
   - 是否具备方案阶段条件
   - 是否具备执行阶段条件
   - 是否具备测试阶段条件
   - 第一轮 next_action
   - 当前是否允许立即进入执行阶段
7. 在完成上述检查前，不得进入实现阶段
```

## 六、使用过程中的自动能力

### 会自动做的事

- 目录模式识别
- first-run 引导
- peer-agent 检查
- 首轮接管盘点
- 治理骨架初始化
- 交接文档与状态文件模板生成
- 任务卡与 dispatch payload 生成
- 恢复摘要生成
- 门禁检查

### 需要调度后完成的事

- 真实方案补完
- 真实业务代码实现
- 真实测试补齐
- 真实审批意见
- 最终终审结论

## 七、与部门 agent 的关系

当 OpenClaw 中存在真实部门 agent 时，本技能会优先调度它们，而不是长期自己模拟所有部门。

当前映射：

- `neige` = 内阁
- `duchayuan` = 都察院
- `libu2` = 吏部
- `hubu` = 户部
- `gongbu` = 工部
- `bingbu` = 兵部
- `libu` = 礼部
- `xingbu` = 刑部

## 八、调度方式

### `sessions_spawn`

适合：

- 一次性任务
- 并行执行
- 测试轮次
- 审查轮次

默认会倾向：

- `cleanup: "delete"`

### `sessions_send`

适合：

- 延续已有 session
- 内阁持续补方案
- 审查返修跟进
- 长线程协作

## 九、常见误区

### 1. 技能安装后会不会立刻自动开发

不会。  
它会先做引导、检查、盘点和分流。

### 2. 技能安装后会不会自动写完所有测试

不会凭空全部写完。  
它会自动生成骨架、模板、门禁和调度要求，然后推动兵部补测试。

### 3. 技能目录是不是项目目录

不是。  
技能目录只是技能包本体；只有目标项目目录才会被接管并写入治理文件。
