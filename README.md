# 司礼监总调度技能

`sili-jian-orchestrator` 是一个面向 OpenClaw 的项目治理型技能。

它不是普通的编码提示词，而是把“司礼监 / 总调度 Agent”制度封装成一套可安装、可恢复、可调度、可检查、可落地的技能包。

这个技能主要用于以下场景：

- 接管做到一半的项目
- 恢复暂停的项目或新会话
- 给项目补齐治理骨架
- 将模糊需求先转成可执行方案，而不是直接开发
- 将新增功能接入已有治理体系
- 在 OpenClaw 中通过司礼监调度内阁、都察院、六部协作推进

## 核心能力

安装并调用本技能后，它会优先按“治理优先”的方式工作，而不是直接进入开发。

它能够：

- 识别当前目录到底是技能目录还是项目目录
- 首次启动时先执行 first-run 引导
- 检查 OpenClaw 中司礼监与同级 agent 是否已就绪
- 对项目执行首轮接管检查
- 初始化项目治理骨架
- 维护状态文件、交接文档、恢复入口、报告模板
- 生成任务卡并构造 `sessions_spawn` / `sessions_send` 调度 payload
- 校验测试、审批、终审、发布门禁
- 支持恢复、继续推进与长期维护

## 与原始提示词的关系

这个仓库不是把原始长提示词简单复制进去，而是把提示词制度拆成了以下几层：

- `SKILL.md`：技能入口、触发边界、首次引导、总流程
- `references/`：制度规则、调度策略、恢复协议、审批规则
- `assets/project-skeleton/`：目标项目治理模板
- `assets/output-templates/`：接管结果、恢复摘要、任务卡等输出模板
- `scripts/`：检查、引导、接管、恢复、门禁、派发等确定性脚本

这样做的优点是：

- 更稳定
- 更适合长期演进
- 更适合 OpenClaw 的技能加载方式
- 不需要每次都把超长提示词整体塞进上下文

## 角色映射

本技能已与 OpenClaw 中的真实部门 agent 对齐：

- `neige` = 内阁
- `duchayuan` = 都察院
- `libu2` = 吏部
- `hubu` = 户部
- `gongbu` = 工部
- `bingbu` = 兵部
- `libu` = 礼部
- `xingbu` = 刑部

请特别注意：

- `libu2` 是 **吏部**
- `libu` 是 **礼部**

这两者在制度化调度里承担完全不同的职责，不能混用。

## 首次使用会发生什么

第一次明确调用本技能时，它会先执行 first-run 引导，而不是直接开发。

首次引导会输出：

- 当前目录模式
- peer-agent 是否就绪
- 当前环境意味着什么
- 最安全的下一步
- 建议下一条提示词

如果当前目录已经是项目目录，它才会继续进入首轮接管检查。

## 这个技能会自动做什么

### 会自动生成或维护

- 项目治理骨架
- `ai/state/*`
- `ai/reports/*`
- `ai/handoff/*`
- `tests/*`
- `workflows/*`
- 首轮接管结果
- 恢复摘要
- 任务卡
- dispatch payload
- 门禁检查结果

### 会自动推动

- 内阁方案补全
- 六部执行分流
- 兵部测试补齐
- 都察院终审
- 状态与交接更新

### 不会凭空完成

- 全部真实业务代码
- 全部真实测试实现
- 全部审批意见内容
- 所有终审结论

它会搭好治理、调度、门禁与恢复框架，然后通过司礼监去调度对应部门完成真实工作。

## 推荐的首次调用方式

在 OpenClaw 中对目标项目目录使用：

```text
使用 $sili-jian-orchestrator 对当前目录执行首次启用引导。

要求：
1. 先不要直接开发
2. 先判断当前目录是技能目录、项目目录还是未知目录
3. 检查 OpenClaw 中司礼监与同级部门 agent 是否已就绪
4. 如果当前目录是项目目录，再继续检查治理状态、tests、workflows、状态文件、恢复入口
5. 输出首次引导结果，至少包含：
   - 当前目录模式
   - peer-agent 是否就绪
   - 当前环境意味着什么
   - 最安全的下一步
   - 建议我下一条输入什么提示词
6. 如果当前目录已经是项目目录，再继续输出首轮接管结果
7. 在完成上述检查前，不要进入实现阶段
```
